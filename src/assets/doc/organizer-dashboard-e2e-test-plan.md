# Market Day 主辦方專區與後台完整 E2E 測試計畫

最後更新：2026-07-22

## 1. 測試目標

本文件規劃主辦方公開專區及登入後的完整後台功能 E2E 測試，驗證公開頁面導覽、響應式操作、角色權限及 Real API 能共同完成主辦方使用流程。

測試範圍包含：

> 主辦方公開專區 → 註冊／登入入口 → 主辦方資料 → 後台首頁 → 通知中心 → 活動管理 → 報名審核 → 收款與退款 → 攤位管理 → 設備管理 → 帳務管理

本測試不是 Smoke，也不加入每次 push 的 `e2e:smoke`。完整測試會建立及修改活動、報名、付款與退款資料，只能由開發者手動執行。

帳號註冊、錯誤密碼、忘記密碼、Google 登入、修改密碼、Google 綁定及註銷帳號，請參考 [Market Day 帳號 E2E 測試計畫](./auth-account-e2e-test-plan.md)。跨角色成功主流程請參考 [Market Day 活動主流程 E2E 測試計畫](./event-main-flow-e2e-test-plan.md)。

## 4. 測試資料規則

- 使用專用 E2E 主辦方、管理員及至少兩個攤主帳號。
- ORG-12 使用 `E2E_EMPTY_ORGANIZER_EMAIL` 指定的專用主辦方帳號；密碼沿用 `E2E_ORGANIZER_PASSWORD`，該帳號不可參與活動或報名流程，以維持通知為空。
- 主辦方、活動、品牌、商品、場地、設備與報名內容使用接近實際營運的中文資料；活動以自然的市集主題組合避免重複。
- 核准及拒絕必須使用不同報名，避免同一筆資料發生互斥狀態。
- 退款必須使用本次執行新建立且已完成 Sandbox 付款的交易。
- 不可使用主流程已完成的活動或付款進行刪除、下架、拒絕與退款。
- 刪除只允許操作本次建立、尚未送審的測試草稿。
- 下載報表需監聽下載事件，至少驗證檔名、檔案大小大於零及回應成功。
- 無法安全刪除的資料以 API 回傳的內部 ID 與測試報告追蹤；timestamp、UUID 與 `E2E` 不得出現在展示名稱。
- 測試不可連接正式資料庫或正式藍新商店。

## 5. 執行規劃

目前指令：

```json
{
  "scripts": {
    "e2e:organizer:portal": "playwright test e2e/organizer/frontend --headed --workers=1",
    "e2e:organizer": "playwright test e2e/organizer --grep-invert \"@sandbox-refund|@profile-reset\" --headed --workers=1",
    "e2e:organizer:backend": "playwright test e2e/organizer/backend --grep-invert \"@sandbox-refund|@profile-reset\" --headed --workers=1",
    "e2e:organizer:profile-guard": "playwright test e2e/organizer/backend/profile-incomplete.spec.ts --grep @profile-reset --headed --workers=1",
    "e2e:organizer:deposit-refund": "playwright test e2e/organizer/backend/deposit-refund.spec.ts --grep @deposit-refund --headed --workers=1",
    "e2e:organizer:refund": "playwright test e2e/organizer/backend/refund.spec.ts --grep @sandbox-refund --headed --workers=1"
  }
}
```

執行方式：

```powershell
# 公開主辦方專區
npm run e2e:organizer:portal

# 主辦方公開專區與一般後台案例
npm run e2e:organizer

# 只執行一般後台案例
npm run e2e:organizer:backend
```

ORG-13 會先由開發者將專用主辦方 Profile 暫時清空，再獨立執行以下指令；測試通過時會立即填回完整的真實風格資料：

```powershell
npm run e2e:organizer:profile-guard
```

退款必須先以本次新付款設定環境變數，再獨立執行：

```powershell
$env:E2E_REFUND_APPLICATION_ID="本次新付款的 applicationId"
npm run e2e:organizer:refund
```

退款失敗重試案例另需設定 `E2E_REFUND_FAILED_APPLICATION_ID`。這筆資料必須確實處於退款失敗，不能拿已退款交易重複使用。

使用單一 worker，避免共用主辦方資料、活動狀態、付款與退款交易互相衝突。藍新 Sandbox 可能拒絕 headless Chromium，因此完整主辦方測試固定使用 headed 模式。

執行前必須確認：

- 前端、後端及測試資料庫已啟動。
- `.env.e2e.local` 已設定主辦方、空通知主辦方、管理員及攤主測試帳密。
- 測試帳號已存在且 Email 驗證完成。
- 後端使用藍新 Sandbox 商店設定。
- NotifyURL 與 ReturnURL 是目前有效、可由外網連線的 HTTPS 網址。
- 使用可正常開啟藍新 Sandbox 的網路環境。

## 6. 與其他 E2E 的分工

- `auth-*.spec.ts`：帳號註冊、登入例外、Email、Google、密碼及註銷。
- `event-main-flow.spec.ts`：跨角色成功主流程及真實付款、選位與品牌公開。
- `e2e/organizer/backend/*.spec.ts`：主辦方後台完整功能、狀態分支、退款與報表。
- `e2e/organizer/frontend/organizer-portal.spec.ts`：主辦方公開專區首頁、關於我們、Header、入口導向與 RWD。

共用操作應抽成 helper，例如登入、建立唯一活動、建立報名、完成付款及等待狀態；不可複製整段主流程程式到每一支測試檔。

## 7. 完成標準

主辦方後台完整 E2E 只有在以下條件全部符合後，才能標示為已完成：

- 本文件列出的測試案例全部實作並通過。
- 所有狀態修改均使用 Real API，不使用前端 Stub 假裝成功。
- 活動送審、撤回、發布、下架及刪除草稿重新整理後仍保持正確狀態。

- 真實藍新 Sandbox 退款完成，且不允許重複退款。
- 攤位、設備、用電與車牌資料和報名內容一致。
- 設備及帳務報表可以成功下載且不是空檔案。
