# AI 輔助 E2E 測試規劃

## 一、規劃目標

本階段先以「主辦方登入」作為 E2E 測試的第一個展示流程，目標是先確認 Playwright 可以成功啟動瀏覽器、自動操作畫面、呼叫後端登入 API，並驗證登入成功與登入失敗兩種情境。

目前暫時不做後端單元測試，也不先測註冊、Email 驗證、活動建立或報名流程。第一階段重點是讓 E2E 測試先跑起來，作為 AI 輔助測試展示的基礎。

## 二、什麼是 E2E 測試

E2E 是 End-to-End Testing，也就是端對端測試。它會模擬真實使用者操作系統，從前端畫面開始，經過後端 API 與資料庫，確認一整條功能流程是否可以正確完成。

以本專題的主辦方登入為例，E2E 測試會驗證：

```text
打開主辦方登入頁
→ 輸入 email 與密碼
→ 點擊登入
→ 呼叫 Spring Boot 登入 API
→ 登入成功後進入主辦方後台
```

這種測試不是只測單一按鈕，而是測整個登入流程是否真的可以被使用者完成。

## 三、什麼是 Playwright

Playwright 是一套瀏覽器自動化測試工具，可以用程式控制瀏覽器，自動執行點擊、輸入、跳轉、等待 API 回應、檢查畫面文字等操作。

本專題會使用 Playwright 來執行 E2E 測試，讓系統自動模擬使用者完成登入流程。

Playwright 可以自動完成：

1. 打開瀏覽器。
2. 前往指定頁面。
3. 輸入帳號與密碼。
4. 點擊登入按鈕。
5. 檢查是否成功進入後台。
6. 登入失敗時檢查錯誤訊息是否顯示。

## 四、Playwright 安裝方式

### 1. 進入 Angular 專案資料夾

```bash
cd D:\maisy\angular_works\market_day
```

### 2. 安裝 Playwright

```bash
npm init playwright@latest
```

安裝時可以選擇：

```text
TypeScript：Yes
Tests folder：e2e
GitHub Actions：No
Install browsers：Yes
```

目前 GitHub Actions 可以先選 No，因為第一階段先在本機執行測試即可。

### 3. 安裝完成後會產生的檔案

安裝後專案中通常會出現：

```text
playwright.config.ts
e2e/
```

其中：

| 檔案或資料夾 | 用途 |
| --- | --- |
| `playwright.config.ts` | Playwright 設定檔 |
| `e2e/` | 放置 E2E 測試程式 |

## 五、Playwright 執行方式

### 1. 啟動前端 Angular

先開第一個終端機，執行：

```bash
npm start
```

確認前端可以在以下網址開啟：

```text
http://localhost:4200
```

### 2. 啟動後端 Spring Boot

再確認後端 Spring Boot 已經啟動，例如：

```text
http://localhost:8080
```

或依照目前後端實際設定的 port 為準。

### 3. 執行 E2E 測試

開第二個終端機，執行：

```bash
npx playwright test
```

這會在背景自動執行測試。

### 4. 顯示瀏覽器操作畫面

如果要讓組員或老師看到瀏覽器真的在自動操作，可以執行：

```bash
npx playwright test --headed
```

### 5. 慢速展示模式

如果覺得瀏覽器跑太快，可以使用慢速模式：

```bash
npx playwright test --headed --slow-mo=700
```

這樣可以更清楚看到 Playwright 自動輸入帳密、點擊登入、檢查結果。

### 6. 查看測試報告

測試完成後可以執行：

```bash
npx playwright show-report
```

報告中可以看到：

1. 哪些測試通過。
2. 哪些測試失敗。
3. 失敗原因。
4. 錯誤截圖或執行紀錄。

## 六、目前測試範圍

目前先測試以下兩條流程：

1. 主辦方登入成功
2. 主辦方登入失敗

這兩條流程範圍較小，但可以證明 E2E 測試能夠驗證前端畫面、後端 API、登入狀態與路由跳轉。

## 七、測試案例一：主辦方登入成功

### 目的

確認使用正確帳號與密碼時，主辦方可以成功登入系統，並進入主辦方後台。

### 測試流程

1. 打開主辦方登入頁。
2. 輸入正確的主辦方 email。
3. 輸入正確密碼。
4. 點擊「登入」按鈕。
5. 等待登入 API 回應。
6. 確認頁面跳轉到主辦方後台。
7. 確認後台畫面有出現主辦方 dashboard 相關內容。

### 預期結果

- 登入成功。
- 網址進入 `/organizer/dash-board` 或 `/organizer/dash-board/home`。
- 畫面顯示主辦方後台。
- 沒有出現錯誤訊息。

## 八、測試案例二：主辦方登入失敗

### 目的

確認使用錯誤密碼時，主辦方不能登入系統，且畫面會顯示錯誤提示。

### 測試流程

1. 打開主辦方登入頁。
2. 輸入正確的主辦方 email。
3. 輸入錯誤密碼。
4. 點擊「登入」按鈕。
5. 等待登入 API 回應。
6. 確認沒有跳轉到主辦方後台。
7. 確認仍停留在登入頁。
8. 確認畫面顯示後端回傳的錯誤訊息。

### 預期結果

- 登入失敗。
- 不會進入 `/organizer/dash-board`。
- 仍停留在 `/organizer/login`。
- 畫面出現錯誤提示，例如「帳號或密碼錯誤」。

## 九、目前測試重點

| 測試項目 | 驗證內容 |
| --- | --- |
| 前端登入畫面 | email、密碼欄位與登入按鈕是否可操作 |
| 後端登入 API | 正確帳密與錯誤帳密是否回傳正確結果 |
| 路由跳轉 | 登入成功後是否進入主辦方後台 |
| 錯誤處理 | 登入失敗時是否停留在登入頁並顯示錯誤訊息 |
| E2E 可視化展示 | Playwright 是否能以瀏覽器畫面展示自動操作流程 |

## 十、目前需要準備的資料

要讓主辦方登入 E2E 測試可以執行，需要先準備：

| 項目 | 說明 |
| --- | --- |
| 前端網址 | 例如 `http://localhost:4200` |
| 後端網址 | 例如 `http://localhost:8080` |
| 主辦方測試 email | 用於登入成功測試 |
| 主辦方正確密碼 | 用於登入成功測試 |
| 錯誤密碼 | 用於登入失敗測試 |
| 錯誤訊息顯示區塊 | 用來確認登入失敗時畫面有顯示錯誤 |

建議在登入頁關鍵元素加上 `data-testid`，讓 Playwright 測試更穩定：

```html
<input data-testid="organizer-login-email" />
<input data-testid="organizer-login-password" />
<button data-testid="organizer-login-submit">登入</button>
<div data-testid="organizer-login-error">
  {{ errorMessage }}
</div>
```

Playwright 測試時即可使用：

```ts
await page.getByTestId('organizer-login-email').fill('organizer@test.com');
await page.getByTestId('organizer-login-password').fill('Test123456');
await page.getByTestId('organizer-login-submit').click();
```

## 十一、未來規劃流程

### 1. 活動建立與發布流程

角色：主辦方、管理員

流程：

```text
主辦方登入
→ 建立活動
→ 上傳攤位配置圖
→ 送出審核
→ 管理員登入
→ 審核通過
→ 主辦方發布活動
```

說明：

這條流程代表活動從建立到可以公開報名，是活動生命週期的起點。

### 2. 攤主報名成功流程

角色：攤主、主辦方

流程：

```text
攤主登入
→ 查看可報名活動
→ 填寫報名申請
→ 送出報名
→ 主辦方登入
→ 審核通過
→ 攤主看到待付款 / 報名成功
```

說明：

這條流程是整個系統最重要的商業流程，用來驗證攤主報名與主辦方審核是否可以正常完成。

### 3. 重複報名阻擋流程

角色：攤主

流程：

```text
攤主已經報名某活動
→ 再次進入同一活動報名
→ 系統阻擋或提示已報名
```

說明：

這條流程用來驗證例外情境，確認系統可以防止同一攤主對同一活動重複報名。

### 4. 審核不通過流程

角色：攤主、主辦方

流程：

```text
攤主送出報名
→ 主辦方審核不通過
→ 攤主查看申請紀錄
→ 狀態顯示審核未通過
```

說明：

這條流程用來測試報名審核的失敗分支，確認主辦方審核不通過時，攤主端會顯示正確狀態。

### 5. 報名截止流程

角色：攤主、系統

流程：

```text
活動報名截止
→ 攤主嘗試報名
→ 系統不允許報名
```

說明：

這條流程用來測試時間規則，確認報名截止後系統不允許攤主再送出報名申請。

## 十二、AI 輔助測試展示方向

未來可將活動流程圖與系統功能描述交由 AI 分析，產生完整測試案例清單，再挑選高風險或核心流程使用 Playwright 執行 E2E 測試。

展示時可以分成三個階段：

1. AI 根據流程產生測試案例。
2. Playwright 自動執行核心 E2E 測試。
3. 系統產生測試結果與錯誤分析報告。
