# Market Day 管理員後台 E2E 測試計畫

最後更新：2026-07-22

## 1. 測試目標

本文件規劃管理員後台專屬功能的 E2E 測試，驗證管理員能透過 Real API 完成以下操作，並確認相關通知、待辦數字與操作紀錄同步正確：

> 總覽儀表板與通知中心 → 活動審核分支動作（要求補件、下架審核同意/駁回）→ 通知主辦方款項結清 → 使用者管理（停用/恢復帳號）→ 操作紀錄查詢

管理員登入本身、帳密錯誤等帳號案例，請參考 [Market Day 帳號 E2E 測試計畫](./auth-account-e2e-test-plan.md)。活動建立、審核通過、地圖建置、發布、報名、付款、選位等跨角色主流程，請參考 [Market Day 活動主流程 E2E 測試計畫](./event-main-flow-e2e-test-plan.md)；本文件只涵蓋 main-flow 沒有涵蓋的管理員分支動作與管理頁面。共用安裝方式與執行規範請參考 [E2E 測試共用指南](./e2e-ai-test-plan.md)。

## 2. 測試範圍

| 身分 | 測試責任 |
| --- | --- |
| 管理員 | 登入、查看總覽與通知中心、要求補件、下架審核同意/駁回、通知款項結清、使用者管理（搜尋、詳細頁、停用/恢復帳號）、操作紀錄查詢 |
| 主辦方 | 使用既有 E2E 主辦方帳號建立、送審、重新送審、發布活動；申請下架（含被駁回後重新申請） |
| 目標主辦方／目標攤主 | 使用專用測試目標帳號，被管理員停用及恢復帳號，並驗證停用期間無法登入 |

管理員只需要建立兩個一次性測試活動：

- **活動 A**：走「補件 → 重新送審 → 審核通過 → 地圖建置 → 發布 → 下架審核（先駁回、再核准）」完整分支流程，時間設定比照主流程（活動時間為未來），測試結束時活動 A 最終狀態為「已下架」。
- **活動 C**：只走到「發布」，但起訖時間設定在測試執行後數分鐘內結束，用來讓「款項未結清」狀態能在測試時限內自然出現，不使用假資料或後端直接寫庫。

## 4. 實作範圍

第一版使用 Real API 完成 ADMIN-01～29 全部項目，不使用 UI Stub。管理員、主辦方沿用主流程既有的共用 E2E 測試帳號；目標主辦方、目標攤主使用專用帳號（見第 6 節），只用於停用/恢復帳號測試，不影響其他測試共用的登入帳號。

## 5. Playwright 腳本規劃

實際測試檔：`e2e/admin-management.spec.ts`

比照 `event-main-flow.spec.ts` 的作法，使用一個測試搭配 `test.step()`，以單一 worker 執行，測試名稱加上 `@admin-management` 標籤：

```ts
test('@admin-management 管理員後台補件、下架審核、款項結清、使用者管理與操作紀錄', async ({ browser }) => {
  test.setTimeout(10 * 60 * 1000);

  await test.step('ADMIN-01～04 管理員登入、總覽、通知中心', async () => {});
  await test.step('ADMIN-05～07 活動 A 建立、要求補件、重新送審', async () => {});
  await test.step('ADMIN-08～10 活動 A 審核通過、地圖建置、發布', async () => {});
  await test.step('ADMIN-11～14 活動 A 下架審核（駁回後核准）', async () => {});
  await test.step('ADMIN-15～17 活動 C 短時間窗與款項結清', async () => {});
  await test.step('ADMIN-18～23 目標主辦方帳號管理', async () => {});
  await test.step('ADMIN-24～28 目標攤主帳號管理', async () => {});
  await test.step('ADMIN-29 操作紀錄查詢', async () => {});
});
```

管理員、主辦方各自保留獨立登入 Context；目標主辦方、目標攤主的登入嘗試各自使用新的 Context，避免互相覆蓋 Token 與 localStorage。

## 6. 測試資料規則

- 管理員、主辦方使用主流程既有的專用 E2E 帳號（`E2E_ADMIN_EMAIL`、`E2E_ORGANIZER_EMAIL`）。
- 新增兩組專用測試目標帳號，只用於本文件的停用/恢復帳號測試，**不可**與其他測試共用的登入帳號混用：

  ```env
  E2E_TARGET_ORGANIZER_EMAIL=目標主辦方測試信箱
  E2E_TARGET_ORGANIZER_PASSWORD=目標主辦方測試密碼
  E2E_TARGET_VENDOR_EMAIL=目標攤主測試信箱
  E2E_TARGET_VENDOR_PASSWORD=目標攤主測試密碼
  ```

  這兩組帳號需事先存在於測試資料庫且能正常登入；測試會實際呼叫停用/恢復 API，並在同一支腳本內於停用後立即恢復，`finally` 中不重複處理（因為恢復本身就是流程的一部分，不是清理動作）。
- 活動 A、活動 C 的名稱加入「時間戳＋UUID」，避免同時或重複執行時資料碰撞；活動封面、地圖圖片重用專案 `src/assets/images/market/cards/` 的固定圖片。
- 活動 A、活動 C 都不涉及攤位報名，因此建立活動時可略過設備租借與額外用電設定（皆為選填）。
- 活動 A 的活動日期沿用主流程慣例設在未來；活動 C 的起訖時間依執行當日動態產生，設定為測試執行後數分鐘內結束，用來讓「款項未結清」狀態能在測試時限內自然出現。
- 應付通知、待辦卡片數字等比對值一律由畫面或 API 當下查到的資料計算，不寫死固定數字。
- 使用獨立測試資料庫或可辨識的 `E2E` 資料，避免污染正式資料。

## 7. 執行方式

本測試會建立及修改資料、實際停用/恢復帳號，使用 `@admin-management` 標籤，排除於每次 push 的 `e2e:smoke`，請手動執行：

```powershell
npm run e2e:admin-management
```

`package.json` 設定：

```json
"e2e:admin-management": "playwright test e2e/admin-management.spec.ts --headed --workers=1"
```

建議執行時機：

- 管理員後台功能完成或修改後。
- 活動審核、下架審核、款項結清或使用者管理 API 契約調整後。
- 合併到主要分支前。

執行前必須確認：

- 前端可正常啟動，後端及資料庫已啟動。
- `.env.e2e.local` 已設定管理員、主辦方既有測試帳密，以及本文件新增的目標主辦方、目標攤主帳密。
- 所有測試帳號都已存在於測試資料庫且能登入。

## 8. 完成標準

- ADMIN-01～29 全部使用 Real API 通過。
- 活動 A 的補件、重新送審、審核通過、地圖建置、發布、下架審核（駁回與核准兩種結果）皆確實寫入後端，且對應通知與首頁待辦數字正確反映每一次狀態轉換。
- 活動 C 在 `eventEndsAt` 實際過去後，讀取到的狀態確實計算為「款項未結清」；通知款項結清後，主辦方收到對應通知。
- 目標主辦方、目標攤主帳號的停用/恢復皆確實寫入後端，停用期間無法登入，恢復後能正常登入，且測試結束時兩個目標帳號都回到「正常」狀態。
- 操作紀錄頁能查到本次測試產生的所有管理員操作，且篩選條件有效。
- 測試可重複執行，不依賴上一次測試留下的活動名稱或日期。

## 9. 最近一次執行結果

- **執行日期**：2026-07-23
- **執行指令**：`npx playwright test e2e/admin-management.spec.ts --headed --workers=1`（與 `npm run e2e:admin-management` 的實際指令相同）
- **結果**：`1 passed (5.6m)`，ADMIN-01～29 全數依序執行並通過。
- **涵蓋範圍**：活動 A 的補件、重新送審、審核通過、地圖建置、發布、下架審核（先駁回再核准）；活動 C 的短時間窗、款項未結清判定、通知款項結清；目標主辦方與目標攤主帳號的停用/恢復與登入驗證；操作紀錄查詢（依活動 A、活動 C、目標主辦方 Email 三種關鍵字搜尋，驗證 `requestRevision`、`eventUnpublishReview`、`notifyEventPayment`、`accountDisabled`、`accountRestored` 皆可查到）。測試結束時兩個目標帳號皆確認回到「正常」狀態。
- **編譯確認**：`npx tsc --noEmit -p tsconfig.app.json` 無錯誤；`npx tsc --noEmit -p tsconfig.spec.json` 有 1 個既有、與本次改動無關的錯誤（`admin-dashboard-home.spec.ts` 的 `systemWarning` 欄位不存在於 `AdminDashboardOverview`），與實作前記錄的 baseline 一致，非本次改動引入。
- **整支分支最終審查**：完成後另做過一次涵蓋全部 12 個任務的整體審查，額外發現並修正一個 race condition——ADMIN-18/24 使用者搜尋跟 ADMIN-29 操作紀錄查詢是同一種問題（進頁面自動觸發的無關鍵字預設查詢，可能被等待邏輯誤接），原本只在 ADMIN-29 修過，這次一併補上並用 `PW_SLOW_MO=0` 重新驗證確認修好，避免掩蓋在 slow-mo 的時間差裡。修正後再次完整執行仍是 `1 passed (5.6m)`。
- **已知風險**（非本次修復範圍，經與使用者確認維持現狀）：ADMIN-18～23／ADMIN-24～28 停用目標主辦方／攤主帳號後、驗證登入被拒與恢復帳號之間沒有失敗保護機制，若中間斷言意外失敗，帳號會卡在停用狀態，需要手動用管理員後台或 API 恢復。

## 10. 後續案例

以下不放進本文件的成功腳本，之後再視需要另外規劃：

- 管理員拒絕活動審核（若後續有獨立「拒絕」動作，而非僅補件）。
- 使用者管理頁刪除帳號等不可逆操作（`@destructive`）。
- 操作紀錄的匯出或大量資料分頁效能。
