# Market Day 活動主流程 E2E 測試計畫

最後更新：2026-07-22

## 1. 測試目標

本文件規劃 Market Day 跨角色的完整活動流程 E2E 測試，驗證前端頁面、角色權限與已串接 API 能共同完成以下流程：

> 主辦方完成資料 → 建立活動 → 管理員審核及完成地圖 → 主辦方發布 → 一般使用者查看 → 攤主完成「我的攤位」 → 帶設備資料報名 → 主辦方核對並核准 → 攤主核對費用、付款及選位 → 主辦方確認收款、設備與攤位 → 一般使用者再次查看公開活動與攤商

帳號註冊、錯誤密碼、忘記密碼、Google 登入等帳號功能不在本文件重複測試，請參考 [Market Day 帳號 E2E 測試計畫](./auth-account-e2e-test-plan.md)。共用安裝方式與執行規範請參考 [E2E 測試共用指南](./e2e-ai-test-plan.md)。

## 2. 測試檔與角色分工

### 2.1 測試檔

| 測試檔 | 負責範圍 | 資料影響 |
| --- | --- | --- |
| `event-main-flow.spec.ts` | FLOW-01～FLOW-34，從活動建立、真實付款、選位到品牌公開 | 建立並修改活動、主辦方資料、攤位、商品、報名、付款及選位資料 |

主流程只維護一支測試檔，使用 `test.step()` 保留 FLOW 編號。四個角色各自使用獨立 Browser Context，避免 Token 與 `localStorage` 互相覆蓋。

### 2.2 角色分工

| 身分 | 測試責任 |
| --- | --- |
| 主辦方 | 登入、填寫主辦方資料、建立活動、送審、發布、核對及核准攤主、確認收款、設備與攤位結果 |
| 管理員 | 登入、審核活動及完成活動攤位地圖 |
| 攤主 | 登入、填寫「我的攤位」與商品、帶設備需求報名活動、核對費用、付款、選位及確認報名結果 |
| 一般使用者 | 不登入，從公開前台查看活動與允許公開的資料 |

登入是主流程的必要前置步驟，只確認正確帳號能登入並進入正確角色後台，不重複帳號測試中的錯誤密碼、登出及 Session 細節案例。

## 4. 實作範圍與外部條件

第一版先使用 Real API 完成以下流程：

1. 主辦方登入、儲存主辦方資料並重新載入確認。
2. 主辦方建立活動並送審。
3. 管理員登入、審核活動並完成攤位地圖。
4. 主辦方發布活動。
5. 一般使用者確認活動與主辦方公開資料正確。
6. 攤主登入、儲存「我的攤位」與固定 E2E 商品並重新載入確認。
7. 攤主帶入租借設備、額外用電與測試車牌後送出報名。
8. 主辦方核對完整報名資料後核准攤主，確認報名進入待付款狀態。
9. 攤主在付款頁確認各項費用與應付總額正確。
10. FLOW-24～34 的自動操作與驗證已寫入同一支腳本，付款成功後會繼續完成選位及跨角色確認。

付款與選位不得只用前端 API Stub 假裝完成，因為 Stub 不會更新後端資料庫，後續選位仍會因為報名尚未付款而失敗。FLOW-24～FLOW-31 及 FLOW-34 必須採用下列其中一種方式後才能完成：

- 使用金流 Sandbox 完成真實測試付款及回呼。
- 由測試環境預先準備一筆已付款的報名資料，從選位流程繼續測試。

### 藍新 Sandbox 測試卡

以下資料只可用於藍新測試環境，不可用於正式金流：

| 欄位 | 測試資料 | 腳本處理方式 |
| --- | --- | --- |
| 信用卡號 | `4000-2211-1111-1111` | 腳本輸入時移除連字號，支援一次付清與分期付款 |
| 有效年月 | 任一未來年月 | 腳本以執行當日加兩年動態產生 `MMYY`，分離欄位則分別填入 `MM`、`YY` |
| 背面末三碼 | 任意三碼；腳本使用 `123` | 只作為 Sandbox 測試安全碼 |

測試卡資料直接定義於 `e2e/event-main-flow.spec.ts`，不需要放入 `.env.e2e.local`；Merchant ID、HashKey、HashIV 與回呼網址仍由後端測試環境設定提供。

### 藍新 Sandbox 執行限制

- 2026-07-22 已使用手機網路分享、有畫面的 Chromium 與 HTTPS 公開回呼網址，完成真實 Sandbox 付款及回呼。
- 藍新 Akamai 仍會拒絕 Playwright 的 headless Chromium，因此主流程指令固定使用 headed 模式；這是第三方測試站限制，不代表 Market Day 付款 API 失敗。
- Quick Tunnel 網址每次重開可能改變。執行前必須把目前的 HTTPS 網址同步設定到後端 NotifyURL 與 ReturnURL，並重新啟動後端。

品牌公開流程已完成：後端每分鐘處理已過報名截止時間的活動，取消未付款報名、替已付款未選位者自動選位，完成後進入 `FINAL_REVIEW` 並寫入 `brands_public_at`。公開 API 只有在公開時間已到時才回傳品牌與地圖。

品牌公開與選位時間的自動測試範圍：

- 截止時間前可以選位；同一天超過完整截止時間後不能選位。
- 未付款報名取消，已付款但未選位者自動分配。
- 自動選位完成後活動進入 `FINAL_REVIEW` 並寫入 `brands_public_at`。
- `brands_public_at` 為空或仍在未來時，公開 API 不回傳品牌與地圖；時間到達後才公開。
- 狀態明確區分「最終名單確認中」與「品牌已公開」。
- 主流程使用短時間報名區間，FLOW-32～34 會輪詢公開 API，確認 `brandsPublic=true` 後再驗證公開地圖與品牌。

後端完整資料庫流程另由 `AutomaticStallAssignmentFlowIT` 驗證，並使用專用的 `MarketDayDB_Test`：

- 多位已付款攤主會取得不同攤位，未付款報名會被取消。
- 分配完成後會寫入 `FINAL_REVIEW`、`brands_public_at`，公開 API 可讀到地圖與品牌。
- 相同排程重複執行不會重複分配或重複建立通知。
- 攤位不足時，同一攤位不會分配給兩位攤主，活動維持等待處理且不提前公開品牌。
- 2026-07-22 最新完整執行結果：`1 passed (7.6m)`；使用不重複且接近正式市集內容的虛構資料，完成 FLOW-01～34 與真實 Sandbox 付款。

2026-07-22 已完成同一支瀏覽器腳本的 FLOW-01～34，不再分成付款前後兩段。

## 6. 測試資料規則

- 使用已驗證且能進入各角色後台的專用 E2E 主辦方、管理員及攤主帳號；首次登入引導不在本腳本重複測試，主流程會以固定資料覆寫並驗證主辦方資料及攤主「我的攤位」。
- 主辦方資料使用真實風格的主辦單位、聯絡人、地址及服務時間，但不使用組員或第三人的真實個資。
- 攤主「我的攤位」使用固定且合理的品牌及商品名稱與測試圖片；腳本儲存前先確認商品是否存在，避免重複新增。
- 品牌頭像、封面、商品圖片與活動地圖重用專案 `src/assets/images/market/cards/` 的固定圖片，不依賴開發者電腦的暫存路徑。
- 活動名稱使用自然的市集主題組合避免重複，例如「日光慢旅生活市集・森野植感午後場」；timestamp、UUID 與 `E2E` 不得出現在展示名稱。
- 主辦方、品牌與商品使用固定且合理的展示名稱；測試以 API 回傳的內部 ID 追蹤資料，不把技術識別碼混入名稱。
- 活動日、報名日及付款期限必須依執行當日動態產生，不可寫死已過期日期。
- 活動至少建立足以完成測試的攤位與可報名名額。
- 報名固定選擇一項租借設備及一項額外用電，並使用每次執行動態產生的測試車牌。
- 測試需記錄攤主送出的日期或攤位類型、設備、用電及車牌，供主辦方詳情與設備管理逐項比對。
- 應付總額由畫面或 API 的費用明細計算，不寫死固定金額；至少核對報名或攤位費、設備費、額外用電及保證金。
- 使用獨立測試資料庫，避免污染正式資料；可辨識性由資料庫內部 ID 與測試報告提供，不犧牲畫面真實度。
- 若系統支援刪除測試草稿或取消測試報名，測試結束後應在 `finally` 清理；不能安全清理時，使用自然且不重複的活動名稱供人工辨識。
- 不可與帳號 Smoke 共用會被修改密碼或註銷的測試帳號。

## 7. 執行方式

主流程會建立及修改大量資料，使用 `@main-flow` 標籤，並排除於每次 push 的 `e2e:smoke` 與帳號非破壞性 `e2e:full`，請手動執行：

```powershell
npm run e2e:main-flow
```

## 8. 一般測試與老師 Demo

主流程只維護一支測試腳本：

```text
e2e/event-main-flow.spec.ts
```

一般測試與老師 Demo 執行相同流程、相同 API 及相同驗證；兩者只在瀏覽器是否顯示、操作速度及錄影用途上不同，不另外複製第二支測試腳本。

| 模式 | 指令 | 瀏覽器 | 速度 | 用途 |
| --- | --- | --- | --- | --- |
| 一般主流程 | `npm run e2e:main-flow` | Headed／全螢幕 | 正常速度 | 開發驗證、合併前檢查及測試報告；避開藍新對 headless 的阻擋 |
| 老師 Demo | `npm run e2e:main-flow:demo` | Headed／全螢幕 | 每個操作約延遲 300～500ms | 專題展示及螢幕錄影 |

目前的 `package.json` 指令：

```json
{
  "scripts": {
    "e2e:main-flow": "playwright test e2e/event-main-flow.spec.ts --headed --workers=1",
    "e2e:main-flow:demo": "playwright test e2e/event-main-flow.spec.ts --headed --workers=1"
  }
}
```

`playwright.config.ts` 只會在 npm 指令名稱為 `e2e:main-flow:demo` 時自動啟用 `E2E_DEMO=1`。一般主流程雖然同樣使用 headed 模式，但不會加入展示停頓。Demo 預設 `slowMo` 為 400ms；若 `.env.e2e.local` 有設定 `PW_SLOW_MO`，則以該值為準。

測試程式只在 Demo 模式加入觀看所需的短暫停頓：

```ts
const isDemo = process.env.E2E_DEMO === '1';

async function demoPause(page: Page, milliseconds = 800) {
  if (isDemo) {
    await page.waitForTimeout(milliseconds);
  }
}

await test.step('主辦方發布活動', async () => {
  await publishButton.click();
  await expect(page.getByText('已發布')).toBeVisible();
  await demoPause(page);
});
```

`demoPause()` 只能放在關鍵結果已驗證之後，不能取代 `expect()` 或用來等待 API 完成。一般模式不執行這些停頓，因此不會拖慢開發測試。

### Demo 影片建議

- 使用 OBS 或 Windows 螢幕錄影擷取全螢幕瀏覽器，建議輸出 1920×1080 MP4。
- 影片以 6～8 分鐘為目標，畫面依序呈現主辦方資料、活動建立、審核、發布、公開查看、攤主「我的攤位」、帶設備資料報名、核准、費用核對、付款、選位及跨角色查看結果。
- 關閉桌面通知、瀏覽器翻譯提示及不相關分頁。
- 不錄製終端機、`.env.e2e.local`、帳號密碼、Token、Email 或付款個資。
- 在角色切換時加入簡短字幕，例如「主辦方建立活動」、「管理員審核」、「攤主選位」。
- 主辦方及一般使用者點選攤位時停留約 1 秒，讓觀看者看清楚兩種角色可見資料的差異。
- Playwright 內建錄影可保留為完整測試證據；專題報告播放另外剪輯的 MP4。

建議交付三種成果：

1. 剪輯過並含步驟字幕的 Demo MP4，報告時播放。
2. Playwright 自動產生的完整測試錄影，證明流程由腳本自動執行。
3. Playwright HTML Report，展示各個 `test.step()` 與最終測試結果。

## 9. 完成標準

完整主流程只有在以下條件全部通過後，才能標示為「已完成」。目前 FLOW-01～34 已在同一次真實 Sandbox E2E 執行中完成：

- 三種角色均使用 Real API 登入成功。
- 主辦方資料經 Real API 儲存並重新載入後一致，公開頁只顯示允許公開的主辦方欄位。
- 攤主「我的攤位」、品牌圖片及自然命名的商品經 Real API 儲存並重新載入後一致。
- 活動、審核、地圖、發布、報名、核准、付款及選位狀態確實寫入後端。
- 攤主送出的日期或攤位類型、租借設備、額外用電及車牌，與主辦方報名詳情及設備管理顯示一致。
- 付款頁各項費用加總等於應付總額，完成付款後攤主與主辦方顯示的付款狀態及金額一致。
- 發布、核准、付款及選位完成後重新整理頁面，狀態與資料仍然存在。
- 一般使用者能在發布後及流程完成後兩次看到公開活動。
- 主辦方在活動地圖點選已分配攤位後，看到的攤主與攤位資料正確。
- 一般使用者在公開地圖點選已分配攤位後，能看到正確品牌、品牌圖片及商品，只顯示允許公開的攤商資訊，不顯示 Email、負責人、設備需求、車牌、付款資料等私人資訊。
- 主辦方、攤主與一般使用者看到的公開攤位結果一致。
- 測試可重複執行，不依賴上一次測試留下的活動名稱或日期。

若付款採用預先建立的已付款資料，文件與測試報告必須明確標示為「分段主流程」，不得宣稱金流端到端整合已完成。

## 10. 後續案例

以下屬於主流程的分支及例外情境，第一版完成後再分開規劃，不放進同一支成功主流程腳本：

- 管理員退回活動，主辦方修改後重新送審。
- 管理員拒絕活動。
- 主辦方取消發布或取消活動。
- 主辦方拒絕攤主報名。
- 攤主付款失敗、逾期或取消付款。
- 攤主選到已被其他攤主占用的攤位。
- 退款、退出活動及名額釋放。
- 活動開始或結束後，公開狀態與操作權限是否正確。

## 11. 更新規則

實作或修改主流程測試後，負責人需同步更新：

- FLOW 編號與測試步驟。
- Real API、測試金流或預先建立資料的使用方式。
- 目前完成狀態及阻擋原因。
- 測試帳號、測試資料及清理要求。
- 最近一次完整執行結果。

不得因單一頁面或 API Stub 測試通過，就把跨角色主流程標示為已完成。
